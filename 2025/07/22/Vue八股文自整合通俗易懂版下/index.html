四、路由

2. 路由的hash和history模式的区别

Vue-Router的hash和history模式，主要区别如下：

  1. URL表现
  • hash模式：URL里带 # ，比如 xxx.com/#/page ， # 后面的部分就是hash值。
  • history模式：URL里没有 # ，和正常网址一样，比如 xxx.com/page 。
  1. 原理与页面刷新
  • hash模式：通过监听 onhashchange 事件，改变 # 后面的内容时，不会向服务器发请求，页面不刷新。
  • history模式：用HTML5的 pushState 等API改变URL，同样不刷新页面，但刷新时如果服务器没对应配置，会出现404。
  1. 对后端的依赖
  • hash模式：完全不影响后端，因为 # 后面的内容不会发给服务器，不用担心404。
  • history模式：需要后端配合配置，否则刷新页面可能返回404（因为浏览器会真实请求这个URL）。
  1. 功能差异
  • history模式更灵活，能设置更丰富的URL（同域名下任意URL），还能存额外数据；hash模式只能改 # 后面的部分，能存的数据有限。
  • hash模式兼容性更好，老浏览器（如低版本IE）也支持；history模式依赖HTML5 API，兼容性稍差。

简单说，hash模式带 # 、兼容性好、不用后端配；history模式URL更美观、功能多，但需要后端支持，否则可能404。

404：是一种HTTP状态码，它表示服务器无法根据客户端的请求找到相应的资源（网页）。通常是因为客户端发送给服务器的URL写错了，或者该资源原本存在但已被删除或移动等。

10. 对前端路由的理解

前端路由是单页面应用（SPA）里的一种技术，核心作用是**：在不刷新页面的情况下，让不同内容对应不同的URL，方便用户前进、后退、刷新，还能优化SEO。**

为什么需要它？
SPA只有一个页面，原本无论内容怎么变，URL都不变。用户刷新页面就会回到初始状态，而且搜索引擎也抓不到不同内容。前端路由就是给每个内容片段分配一个唯一URL，解决这些问题。

怎么实现的？
核心是“拦截URL变化，用JS更新内容”：

  • 不让浏览器因为URL变了就去请求服务器（避免刷新）。
  • 自己监听URL的变化（比如hash模式的 # 后面部分变了，或history模式的API操作），然后根据不同URL展示对应的组件/内容。

简单说，前端路由就是前端自己管理URL和内容的对应关系，让单页应用像多页应用一样有“页面感”，但体验更流畅。

五、Vuex

1. Vuex 的原理

Vuex 是 Vue 里管理全局状态的工具，核心是一个“仓库（store）”，把组件共享的数据集中存这里，实现数据统一管理。

流程很简单：

  1. 组件里通过 dispatch 触发 actions （可以处理异步，比如调接口）；
  2. actions 通过 commit 调用 mutations ；
  3. mutations 直接修改 state 里的数据；
  4. state 变化后，组件会自动更新渲染。

简单说就是：组件→actions（处理异步）→mutations（改数据）→state（存数据）→组件更新。

2. Vuex中action和mutation的区别

  • mutation：只能同步操作，是修改 state 的唯一方式，通过 commit 触发。
  • action：可以处理异步（比如请求接口），不能直接改 state ，必须通过 commit 调用 mutation ，通过 dispatch 触发。

一句话：mutation 负责“同步改数据”，action 负责“处理异步，再让 mutation 改数据”。

4. Redux 和 Vuex 有什么区别，它们的共同思想

共同点

两者都是管理全局状态的工具，核心思想一致：

  • 都有一个单一数据源（全局只存一份数据）；
  • 数据变化可预测（改数据有固定流程，方便追踪）；
  • 本质都是把数据从视图中抽离，让状态管理更清晰。

区别

  • 写法简化：Vuex 比 Redux 简单。Redux 用 reducer （需要 switch 判断）处理数据，Vuex 用 mutations 直接改数据，不用写复杂逻辑；
  • 异步处理：Redux 里异步操作要放 action 里，还可能需要中间件（如 thunk）；Vuex 直接在 actions 里写异步，更直观；
  • 自动渲染：Vuex 借助 Vue 的响应式，数据变了组件自动更；Redux 需手动订阅数据变化来触发渲染。

简单说：两者思想相通，都是管全局数据的，但 Vuex 更贴合 Vue 的特性，用起来更简单；Redux 流程更严谨，适合 React 生态。

六、Vue 3.0

1. Vue3.0有什么更新

  1. 数据响应式升级：
    用 Proxy 代替了 Vue2 的 Object.defineProperty ，能直接监听整个对象（包括新增/删除属性、数组索引变化等），解决了 Vue2 里数据监听的各种限制（比如不能监听数组长度变化）。
  2. 性能优化：
    模板里的作用域插槽改成函数形式，父组件不会因为子组件插槽变化而重新渲染，渲染性能更好。
  3. 组件写法更灵活：
    支持类式组件声明，和 TypeScript 结合更方便；还能直接用 setup 函数写逻辑，不用再写 data 、 methods 这种选项式写法。
  4. 新增功能：
    支持多根节点（Fragment）、在 DOM 其他地方渲染组件（Portal），还有更好的 Tree-shaking 优化（打包更小）。

2. defineProperty和proxy的区别

  • Vue2（defineProperty）：只能监听对象的某个属性，新增/删除属性、数组下标变化都监听不到，得用 $set 手动处理。
  • Vue3（Proxy）：直接代理整个对象，不管新增、删除属性，还是数组变化，都能自动监听，更强大、更省心。

简单说，Vue3 主要胜在响应式更完善、性能更好、写法更灵活，对开发者更友好~

七、虚拟DOM

1. 对虚拟DOM的理解？

虚拟DOM简单说就是用JavaScript对象来描述真实的DOM结构。

比如一个真实的

Hello
，虚拟DOM可能长这样： { tag: ‘div’, attrs: { class: ‘box’ }, text: ‘Hello’ } 。

作用：

  1. 提升性能：数据变化时，先对比新旧两个虚拟DOM对象（用diff算法），只把不一样的地方更新到真实DOM上，减少直接操作DOM的次数（DOM操作很耗性能）。
  2. 跨平台：虚拟DOM是JS对象，不依赖浏览器环境，能轻松实现多端渲染（比如网页、手机App、服务端渲染SSR）。
  3. 简化开发：不用手动写操作DOM的代码，框架会自动通过虚拟DOM处理渲染，开发者更专注业务逻辑。

简单说，虚拟DOM就是DOM的“轻量替身”，帮我们高效、跨平台地更新页面。

面试官问起对虚拟DOM的理解时，你可以这样回答，既简洁又能抓住重点：

“虚拟DOM其实就是用JavaScript对象来描述真实的DOM结构，比如一个div标签，会被描述成包含标签名、属性、子元素等信息的对象。

它的主要作用有两个：一是提升性能，数据变化时，先通过diff算法对比新旧虚拟DOM的差异，只更新变化的部分到真实DOM，减少DOM操作的开销；二是方便跨平台，因为虚拟DOM是纯JS对象，不依赖浏览器环境，能轻松支持服务端渲染、移动端等场景。

对开发者来说，不用手动操作DOM，框架会通过虚拟DOM自动处理渲染，让我们更专注业务逻辑。”

这样回答既解释了核心概念，又说明了作用，语言口语化，适合面试场景~

5.Diff算法的原理

Diff算法是虚拟DOM更新时，对比新旧虚拟DOM差异的方法，核心是高效找出变化的部分，原理可以简单总结为：

  1. 先比自身：判断新旧节点是不是同一个（比如标签名、key相同），不是就直接删旧换新。
  2. 再比子节点：
  • 若一方有子节点、一方没有，直接增删子节点。
  • 若双方都有子节点，就用“同层比较”（只比同一层级的子节点，不跨级比），通过key匹配相同子节点，递归比较它们的差异。

这样一来，把原本复杂的对比（时间复杂度O(n³)）简化成了O(n)，效率很高。

简单说，Diff就是“先看自己是不是同一个节点，再高效对比子节点，只更改变化的部分”。

1. src和href的区别

src和href都是用来引用外部的资源，它们的区别如下：

  • src： 表示对资源的引用，它指向的内容会嵌入到当前标签所在的位置。src会将其指向的资源下载并应⽤到⽂档内，当浏览器解析到该元素时，会暂停其他资源的下载和处理，直到将该资源加载、编译、执⾏完毕，所以⼀般js脚本会放在页面底部。（当浏览器解析到 时，会暂停渲染页面剩余内容，先下载 main.js 并执行里面的代码，执行完才继续渲染

    标签内容 ，这就是 src 会“暂停其他资源处理”的体现。）

  • href： 表示超文本引用，建立和当前元素或本文档的链接关系。当浏览器识别到它他指向的⽂件时，就会并⾏下载资源，不会停⽌对当前⽂档的处理。 常用在a、link等标签上。

2. 对HTML语义化的理解

语义化是指根据内容的结构化（内容语义化），选择合适的标签（代码语义化）。根据网页内容的含义（比如“这是导航”“这是主要内容” ），选对应的 HTML 标签，让标签本身“说出”内容是干啥的，而不是全用无意义的

堆页面。

语义化的优点如下：

  • 对机器友好，带有语义的文字表现力丰富，更适合搜索引擎的爬虫爬取有效信息，有利于SEO。
  • 对开发者友好，使用语义类标签增强了可读性，结构更加清晰，开发者能清晰的看出网页的结构，便于团队的开发与维护。

3. DOCTYPE(⽂档类型) 的作⽤

浏览器渲染网页时，DOCTYPE就是给浏览器定规则的：得先搞清楚 “按啥标准解析 HTML/CSS/JS”。。它必须声明在HTML⽂档的第⼀⾏。

  • 写 （HTML5 常用 ），浏览器就按 标准模式（CSS1Compat ）解析，严格遵守 W3C 规范（比如宽高计算、样式渲染更统一 ）；

  • 要是没写 DOCTYPE ，浏览器可能进入 怪异模式（BackCompat ），用老规则、宽松解析（容易出现样式混乱，不同浏览器表现还不一样 ）。

    4. script标签中defer和async的区别

    详细白话：都让 JS 脚本 “不阻塞页面加载”，但执行时机有差异！。网页加载时，遇到 标签（没加 defer/async ），会 暂停加载页面，先下载、执行 JS，完了再继续加载页面（容易让用户看到 “卡一下” ）。加了 defer 或 async ，就能让 JS 边加载页面边下载脚本（不阻塞页面 ），但执行时机不同：

    • async：脚本下载完 立刻执行（不管页面有没有加载完 ），适合广告、统计代码这类 “早执行但不影响页面结构” 的脚本；

    • defer：脚本下载完，等 页面全部加载完 再执行（能保证执行顺序和 DOM 完整 ），适合依赖 DOM 元素的脚本（比如操作页面按钮、渲染内容 ）。

      6. HTML5有哪些更新

    大白话给你拆解下：就是问 HTML5 比以前的 HTML 多了啥新东西，比如：

    • 语义化标签：像 header （网页头部）、 footer （网页底部）、 nav （导航栏）这些，让网页结构更清晰，机器和人都好懂。
    • 媒体标签： audio （音频）、 video （视频），以前得靠插件才能播音频视频，现在 HTML5 自己就支持了，还能设置自动播、循环播这些。
    • 表单新功能：输入框类型变多了，比如 email （专门输邮箱，还能自动验证格式对不对）、 number （只能输数字，还带加减按钮）；还能加提示文字（ placeholder ）、自动聚焦（ autofocus ）这些小功能，用户体验更好。
    • ** canvas、SVG**：用来在网页上画图、做动画的， canvas 更像“像素级”画画， SVG 是矢量图，放大不会糊。
    • Web 存储： localStorage （存的数据关了浏览器也在）、 sessionStorage （关了浏览器就没了），比以前的 cookie 存得更多、更方便。

15. Canvas和SVG的区别

咱把 Canvas 和 SVG 想象成俩“画图工具”，用大白话对比区别：

一、SVG：“智能矢量画图法”

  • 特点：
  • 不挑分辨率：不管放大多少倍（手机、电脑、投影），图都不会糊，因为它是用“数学公式”描述图形（比如画圆就记圆心、半径 ）。
  • 能互动：图形是“活”的 DOM 元素，点一下、移过去能触发事件（比如点 SVG 地图上的城市，弹出介绍 ）。
  • 适合“大区域、少变化”：像谷歌地图（放大缩小不变形）、流程图（节点能点击），但图形太复杂（比如几千个小图形）会变卡。
  • 不适合游戏：因为互动是靠 DOM，频繁动的话性能跟不上。
  • 人话总结：SVG 画的图“放大不糊、能点能互动”，适合做地图、图表这些静态/少动的矢量图。

二、Canvas：“像素级手绘法”

  • 特点：
  • 挑分辨率：放大容易糊，因为它是“一个像素一个像素”画出来的（像用颜料填格子 ）。
  • 没互动（默认）：图形是“死”的像素，想互动得自己写代码判断（比如点一下 Canvas，手动算有没有点到图形 ）。
  • 适合“频繁动、像素级控制”：像游戏（天天酷跑里的角色、子弹）、实时数据可视化（动态变化的波形图 ），因为画得快、能频繁重画。
  • 能存成图片：画完直接导出 PNG/JPG，当普通图片用。
  • 人话总结：Canvas 是“手动像素填色”，适合做游戏、动画这些需要频繁变的动态图，就是互动得自己操心。

三、选谁？看需求！

  • 做地图、流程图、图标（要放大不糊、能点击）→ 选 SVG；
  • 做游戏、动态数据可视化、复杂动画（要频繁动、像素级控制）→ 选 Canvas；

简单说：SVG 像“智能矢量图，能动会响”；Canvas 像“像素手绘板，灵活但得自己搞互动”，按需选就行～

16. head 标签有什么作用，其中什么标签必不可少？

把 HTML 文档当成一本百科全书，head 标签就如同书的目录页。普通读者（网页浏览者）浏览正文（网页内容）时不太在意目录页，但它对书很关键。

  • head 标签作用：指引样式：像目录中标注图表位置一样，head 里的 标签指引浏览器找到 CSS 样式表，让百科全书排版美观，网页更好看。
  • 规划互动：类似目录标注书中互动环节， 标签在 head 里引入 JavaScript 脚本，让读者能参与互动，网页有各种功能。
  • 记录信息：如同目录附带书的基本信息， 标签在 head 中提供网页元信息，帮搜索引擎了解网页。
  • 必不可少的标签： 标签是目录页最上方的书名，在浏览器标签页显示，让用户知道网页核心内容。
  1. 文档声明（Doctype）和 有何作用？严格模式与混杂模式如何区分？它们有何意义？

继续用百科全书比喻，浏览器是负责解读书的工具。

  • 文档声明（Doctype）作用：Doctype 像是百科全书的编写规范说明，告诉浏览器这本书按哪种 HTML 规范写的，浏览器才能正确解读。
  • 作用：这是 HTML5 的声明，告诉浏览器“这本书按 HTML5 规范编写，按此标准解析渲染”。没它，浏览器可能用旧方法，网页显示异常。
  • 严格模式与混杂模式区分：
  • 严格模式：浏览器像严格按规范解读书的工具，按 W3C 标准解析代码，不同浏览器显示一致。
  • 混杂模式：没正确规范说明时，浏览器凭经验解读，不同浏览器显示不同。
  • 意义：
  • 严格模式意义：鼓励按规范写书（写代码 ），书质量高，网页显示效果好且稳定。
  • 混杂模式意义：为兼容早期不规范的书（网页 ），但新创作应避免，用严格模式。

17. 文档声明（Doctype）和<!Doctype html>有何作用? 严格模式与混杂模式如何区分？它们有何意义?

一、文档声明（Doctype）的作用

文档声明就是给浏览器告诉它：当前这个 HTML 文档，是用哪个版本的 HTML 语法写的！浏览器知道版本后，才会用对应的规则解析网页，保证网页显示正常。

二、 的作用
让浏览器开启 “标准模式” ，用 HTML5 的最新规则解析网页。如果不写这句，浏览器会进入 “混杂模式” （用老版本、不统一的规则解析），网页在不同浏览器显示可能变形、错位，所以写 HTML5 网页时，第一行必须加它！

三、严格模式 vs 混杂模式（核心区别）

  • 严格模式（标准模式）：
    浏览器严格按照 W3C 官方标准 解析网页代码。不管用什么浏览器，同一网页显示效果几乎一样，不会乱套。新网页必须用严格模式！
  • 混杂模式（怪异模式）：
    浏览器不按官方标准，而是用 自己“瞎猜”的老规则 解析。不同浏览器“猜”的规则不一样，同一网页在 Chrome 看着正常，在 Safari 可能变形。这模式是为了兼容很多年前的老网站（那时候 HTML 规范不严格），现在写新网页，必须避免！

四、怎么区分当前是哪种模式？

看 文档声明（Doctype）怎么写 ：

  • 如果写了 （HTML5 声明 ）→ 浏览器用 严格模式 解析。

  • 如果没写文档声明，或者写的声明不标准）→ 浏览器用 混杂模式 解析。

19. 渐进增强和优雅降级之间的区别

渐进增强和优雅降级，都是前端开发里为适配不同浏览器/设备，优化用户体验的思路

一、渐进增强（progressive enhancement）

  • 思路：从基础版开始，逐步往上加高级功能
    先保证“最旧/最弱的浏览器/设备”能正常用（比如只显示文字、基础按钮 ），再给“新浏览器/高端设备”加特效（动画、交互、复杂布局 ）。

二、优雅降级（graceful degradation）

  • 思路：从完整版开始，逐步往下兼容旧设备
    先给“最新/最强的浏览器/设备”做最完整、最炫的功能（动画、交互全上 ），再给“旧浏览器/低端设备”砍功能、简化体验，保证能用但不追求完美。

三、一句话总结区别

  • 渐进增强：从弱到强，先保能用，再升级体验（基础 → 高级）
  • 优雅降级：从强到弱，先做完美，再简化兼容（高级 → 基础）

现在前端更推荐“渐进增强”，因为更贴合“内容优先、适配未来”的开发理念～
